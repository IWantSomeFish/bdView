## Конкретно что не так

### 1. Управление моделями — самая большая проблема

Ты создал свой тип `RouteSimilarityModel` в `model/model.types.ts`. Там одна модель,
один тип, всё жёстко под задачу сравнения маршрутов. Расширить это под другой тип
модели невозможно без переписывания.

Должно было быть сделано универсально — чтобы модели разных типов хранились в одном месте,
загружались одной функцией, а параметры обучения передавались через запрос. Примерно так:

```typescript
// Вызов тренировки с параметрами через API
POST /models/train
body: {
  minAppearanceFreq: 0.4,
  maxVariance: 100,
  minSpatialConsistency: 0.3
}
```

У тебя эти параметры захардкожены в нескольких местах: часть в `.env`,
часть прямо в `api.service.ts` (`minRouteSimiliraty: 0.55`, `minCosin: 0.55`),
ещё часть в `trajectory/trajectory.build.ts` (`resolution = 11`). Изменить их
можно только лезя в код.

### 2. Эндпоинт `/models/upload` не работает

В `ModelsTab.tsx` (строка ~42) есть кнопка загрузки модели, которая
делает `POST /models/upload`. Но в `api.routes.ts` такого маршрута нет.
При нажатии на кнопку будет 404. Это не проверялось или что?

---

### 4. Модели сохраняются как JSON-файлы без порядка

После обучения появляются файлы вида `route-model-trained-2026-06-23-14-30.json`,
`route-model-trained-2026-06-23-15-12.json` и так далее. При этом в контроллере
путь прописан жёстко: `MODEL_PATH = './route-model.json'`. То есть эти файлы
просто накапливаются и никак не используются. Нет ни версионирования, ни
возможности загрузить конкретную версию — ни из кода, ни из интерфейса.

---

### 5. TypeScript `any` в нескольких компонентах

В `RoutesPage.tsx` и `SimilarMap.tsx` используется `Set<any>`, `err: any`,
приведения `as any`. Если уже пишешь на TypeScript — пиши на TypeScript.
`any` это способ сказать компилятору «не проверяй это», что убивает половину
смысла использования TS.

## Что нужно было сделать

По заданию три задачи:

**Задача 1** — в `api.service.ts` загрузить модель один раз при старте
(`loadModel(modelJson)`) и использовать `filterStableNetworks()` внутри
`buildOutput` чтобы считать `stableNetworkCount` для каждой калибровки.

**Задача 2** — в `api.controller.ts` добавить три метода: `listModels`,
`uploadModel`, `trainModel` — и зарегистрировать их в `api.routes.ts`.
Параметры обучения (`minAppearanceFreq`, `maxVariance`, `minSpatialConsistency`)
должны приходить из тела запроса, а не быть зашиты в код.

**Задача 3** — убрать заглушки из `ModelsTab.tsx`, чтобы кнопки реально
делали запросы к работающим эндпоинтам.

Всё это — подключение готового кода, не написание нового с нуля.
