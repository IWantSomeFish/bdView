# WiFi Tracker Backend

Это серверная (бэкенд) часть мобильного приложения **WiFi Tracker & Speedometer**. Проект разработан на платформе **Node.js** с использованием **Express** и **TypeScript**. В основе архитектуры лежит паттерн **Layered Architecture (Многослойная архитектура)**, что обеспечивает четкое разделение ответственности, безопасность типов и простоту масштабирования.

---

## 🏗 Архитектура и структура кода

Проект разделен на логические слои. Каждый слой выполняет строго одну задачу, передавая данные сверху вниз — от сетевого запроса к базе данных.

```text
[ Запрос от клиента ] 
          │
          ▼
   routes (Маршрутизация) ──► middlewares (Валидация / Авторизация)
          │
          ▼
   controllers (Обработка запроса и ответа)
          │
          ▼
   services (Бизнес-логика и Расчеты)
          │
          ▼
   repositories (Слой доступа к данным)
          │
          ▼
[ Хранилище данных / База данных ]
```

### Назначение папок

*   **`config/`** — Конфигурация приложения: настройки портов, подключений к базам данных и чтение переменных окружения (`.env`).
*   **`routes/`** — Маршрутизация (API эндпоинты). Определяет доступные URL-адреса и связывает их с нужными контроллерами.
*   **`middlewares/`** — Промежуточное ПО. Выступает в роли фильтров: проверяет авторизацию (например, JWT), валидирует входящие данные и перехватывает ошибки.
*   **`controllers/`** — Управление жизненным циклом запроса. Принимает HTTP-запрос, забирает из него данные, вызывает бизнес-логику из сервисов и отправляет клиенту HTTP-ответ.
*   **`services/`** — «Мозг» приложения. Содержит чистую бизнес-логику, алгоритмы фильтрации сигналов Wi-Fi, расчеты скорости (`speedometer`) и агрегацию метрик.
*   **`repositories/`** — Слой доступа к данным (DAL). Содержит прямые запросы к базе данных или файлам. Изолирует бизнес-логику от особенностей хранения данных.
*   **`types/`** — Глобальные типы и интерфейсы TypeScript. Обеспечивают строгую типизацию данных между всеми слоями.
*   **`data/`** — Локальное хранилище данных. Используется для хранения файлов базы данных (например, SQLite), логов или кэша.
*   **`dist/`** — Папка сборки. Содержит скомпилированный production-ready код на чистом JavaScript, созданный из файлов TypeScript. *Файлы здесь вручную не редактируются.*

---

## 🔄 Жизненный цикл запроса (Пример)

Когда мобильное приложение отправляет на сервер новые замеры скорости или данные сканирования сетей:

1. **`routes`** принимает `POST`-запрос и направляет его по цепочке.
2. **`middlewares`** проверяют, авторизован ли телефон и корректен ли формат присланного JSON.
3. **`controllers`** забирают данные из тела запроса и передают их в слой сервисов.
4. **`services`** обрабатывают телеметрию (считают среднюю скорость, отсекают аномалии сигнала).
5. **`repositories`** сохраняют очищенные данные в базу данных внутри папки `data/`.
6. **`controllers`** отправляет мобильному приложению JSON-ответ об успешном сохранении.

# WiFi Tracker Frontend (Web Client)

Это веб-интерфейс (панель мониторинга) для проекта **WiFi Tracker & Speedometer**. Приложение построено на базе **React**, **TypeScript** и сборщика **Vite**.

Главная особенность фронтенда — интеграция с картами **Leaflet** для визуализации Wi-Fi сетей и использование принципов **Чистой архитектуры / DDD (Domain-Driven Design)**, что изолирует бизнес-логику от фреймворка и UI.

---

## 🏗 Архитектура и структура кода (`src/`)

Код разделен на независимые слои, где зависимости направлены строго извне внутрь (от деталей реализации к бизнес-правилам).

```text
[ Presentation Layer (UI) ] ──► [ Use Cases (Сценарии) ] ──► [ Domain (Бизнес-логика) ]
            │                                                            ▲
            ▼                                                            │
[ Infrastructure Layer (HTTP / API) ] ───────────────────────────────────┘
```

### Разбор слоев автоматизации

*   **`domain/`** — Ядро приложения. Содержит бизнес-сущности (например, интерфейсы `WifiPoint`, `Speedtest`) и интерфейсы репозиториев (`repositories/`), которые описывают, *какие* действия с данными возможны, но не знают, *как* они технически выполняются.
*   **`usecases/`** — Сценарии использования. Чистая логика приложения. Описывает конкретные процессы (например, `GetWifiMapPoints`, `RunSpeedTest`). Они берут данные из абстрактных репозиториев и передают их в UI.
*   **`infrastructure/`** — Слой реализации (детали).
    *   `http/` — Конфигурация сетевых запросов (Axios/Fetch), базовые URL бэкенда, перехватчики (interceptors) токенов.
    *   `repositories/` — Конкретная реализация интерфейсов из слоя Domain. Именно здесь методы делают реальные сетевые запросы к бэкенду (`api.get('/points')`) и возвращают готовые сущности.
*   **`presentation/`** — Слой отображения (все, что связано с React).
    *   `components/` — UI-компоненты. Карты Leaflet, маркеры Wi-Fi сетей, спидометры, графики, таблицы истории.
    *   `hook/` — Кастомные React-хуки для связи компонентов со сценариями (`usecases`), управления состоянием экрана и жизненным циклом компонентов.

### Сервисные папки проекта
*   **`public/`** — Статические файлы, которые копируются в сборку как есть (иконки, favicon, конфигурации).
*   **`data/`** — Локальное окружение. Содержит локальную базу данных `sqlite3`, которая может использоваться для хранения промежуточного состояния, локального тестирования или кэширования.
*   **`web/dist/`** — Папка, куда **Vite** собирает готовый проект. Содержит оптимизированные, сжатые файлы для деплоя на сервер.

## 🤖 Конвейер нейросетей и автокалибровки (Dual-Model ML Pipeline)

Для фильтрации радиоэфира и точного позиционирования бэкенд использует каскад из **двух специализированных нейросетевых моделей** в слое `services/`. Они обрабатывают нормализованные после автокалибровки векторы уровней сигналов RSSI.

```text
[ Входной вектор RSSI ]
           │
           ▼
┌──────────────────────────────────────┐
│  Модель 1: Многослойный перцептрон   │ ──► [ 0: Нестабилен ] ──► (Запрос отклонен)
│         (MLP Классификация)          │
└──────────────────────────────────────┘
           │
     [ 1: Стабилен ]
           │
           ▼
┌──────────────────────────────────────┐
│  Модель 2: Линейный нейрон (SGD)     │ ──► [ Выход: Координаты ]
│          (Регрессия)                 │       (Latitude, Longitude)
└──────────────────────────────────────┘
```

### 1. Архитектура нейросетевых моделей

#### Модель 1: Фильтрация среды (Многослойный перцептрон / MLP)
* **Задача:** Бинарная классификация стабильности радиоэфира.
* **Принцип работы:** На вход подается вектор значений RSSI от окружающих Wi-Fi точек. MLP анализирует скрытые нелинейные зависимости (затухания, интерференцию, шумы).
* **Выходной слой:** Функция активации *Sigmoid*, выдающая `0` (сигнал нестабилен/аномален, расчет координат не имеет смысла) или `1` (сигнал пригоден для позиционирования). Данный подход минимизирует «скачки» маркеров на карте из-за отраженного сигнала.

#### Модель 2: Позиционирование (Линейный нейрон с обучением через SGD)
* **Задача:** Регрессия для предсказания непрерывных числовых величин — географических координат ($Latitude$ и $Longitude$).
* **Принцип работы:** Если Модель 1 подтвердила стабильность сигнала, вектор RSSI передается Линейному нейрону. Модель вычисляет взвешенную сумму сигналов, где веса отражают удаленность от базовых станций.
* **Автокалибровка и Обучение (SGD):** Использование стохастического градиентного спуска (Stochastic Gradient Descent) позволяет модели обучаться в режиме *Online-learning*. При получении эталонных координат от устройства (например, при хорошем сигнале GPS), SGD мгновенно корректирует веса линейного нейрона. Это реализует **автокалибровку**: система «на лету» адаптируется к изменению радиоокружения (появление новых стен, перемещение роутеров) и компенсирует разницу в чувствительности антенн различных смартфонов.

### 2. Математический жизненный цикл замера
1. Приложение отправляет снимок эфира `[BSSID_1: RSSI_1, BSSID_2: RSSI_2, ...]`.
2. **MLP** оценивает качество кадра. Если на выходе `0`, пакет отбрасывается для защиты от зашумления трека.
3. Если на выходе `1`, **Линейный регрессор** рассчитывает точную точку нахождения.
4. Если в пакете присутствуют валидные GPS-координаты со смартфона, запускается шаг обратного распространения ошибки через **SGD** для калибровки весов под текущие условия.

### 3. Управление жизненным циклом и версионированием моделей

Обучение нейросетей изолировано на уровне конкретных инфраструктурных объектов. Запуск процесса осуществляется через API-метод `trainModel` со следующей логикой:

*   **Локализация обучения (`routeId` / `segmentId`):** Модели обучаются не на всем массиве данных, а кастомизируются под конкретные маршруты и сегменты дорог. Это позволяет Линейному нейрону (SGD) точнее настраивать веса расстояний до роутеров, зная геометрию конкретного участка.
*   **Версионирование (`modelVersion`):** Система поддерживает параллельное существование нескольких версий моделей для одного сегмента. При запуске обучения можно указать тег версии для отслеживания прогресса точности (метрики Loss/Accuracy). Если версия не указана, бэкенд автоматически инкрементирует или генерирует уникальный идентификатор.
*   **Процесс вызова (UI/Инференс):**
    Панель администратора отправляет асинхронный запрос `await vm.trainModel(...)`. Бэкенд ставит задачу на обучение в очередь, собирает из локальной БД (`sqlite3`) исторические данные по указанному `segmentId`, нормализует RSSI, проводит обучение каскада моделей (MLP и SGD) и сохраняет веса, привязанные к версии и сегменту. После успешной отправки команды состояние UI сбрасывается для защиты от повторных триггеров.

## 🤖 Машинное обучение и Автокалибровка (ML & Auto-Calibration)

Проект использует каскад нейросетевых моделей для фильтрации радиоэфира, калибровки датчиков и сглаживания координат. Процесс обучения изолирован на уровне маршрутов/сегментов и поддерживает следующие архитектурные конфигурации:

### 1. Типы обучаемых моделей (`modelType`)

Система поддерживает 5 специализированных режимов обучения под разные задачи позиционирования:

*   `stable_network_filter` (Модель 1: MLP Классификатор) — Базовое обучение Многослойного перцептрона для бинарного разделения радиоэфира на стабильный (`1`) и зашумленный (`0`).
*   `stable_network_filter_aligned` — Обучение классификатора MLP с предварительной аппаратной нормализацией уровней сигналов (RSSI) для компенсации разницы чувствительности антенн разных смартфонов.
*   `stable_network_filter_aligned_trace` — Продвинутое обучение фильтра стабильности с выравниванием сигналов и привязкой к жесткому эталонному следу (Trace) движения по сегменту дороги.
*   `gps_aligned_smoothing` (Модель 2: Линейный регрессор / SGD) — Линейный нейрон с обучением через стохастический градиентный спуск, осуществляющий пространственное выравнивание предсказанных координат относительно реальной сетки дорог.
*   `gps_smoothing` (Дефолтная модель) — Алгоритм регрессии для фильтрации «скачков» GPS-приемника и построения плавной линии трека.

---

### 🔄 Жизненный цикл и асинхронный пайплайн обучения

Вызов метода `await useCases.trainModel(payload)` запускает асинхронную цепочку на бэкенде:

```text
[ Выбор параметров UI ] ──► [ Смена статуса UI ] ──► [ Вызов UseCase ]
                                                           │
                                                           ▼
[ Обновление списков моделей ] ◄── [ Вывод результата ] ◄── [ Расчет на бэкенде ]
```

1. **Инициализация:** На основе переданного `payload` интерфейс определяет тип модели и мгновенно выводит пользователю детализированный статус (например: *"Запуск обучения фильтра стабильных сетей с выравниванием..."*).
2. **Выполнение сценария (Use Cases):** Запрос передается в слой чистой бизнес-логики (`useCases.trainModel`), который собирает из базы данных (`sqlite3`) сэмплы для конкретного `routeId` и `segmentId`.
3. **Тренировка и логирование:** По завершении обучения бэкенд возвращает объект `result`, содержащий сгенерированную **версию модели** (`result.version`) и **количество сэмплов** (`result.sampleCount`), на которых она училась. UI выводит отчет: `Модель v1.0.2 обучена на 1540 сэмплах.`
4. **Релоадинг контекста:** После успешного обучения фронтенд вызывает параллельно методы `await loadModels()` и `await loadAiModels()`. Это необходимо для того, чтобы новые веса нейросетей мгновенно подгрузились в память сервера/клиента и стали доступны для инференса «на лету» без перезапуска приложения.


ФРОНТЕНД (Клиент)
1. UI Компонент         ──► Нажатие «Обучить» -> вызов vm.trainModel(...)
2. View Model (vm)      ──► Установка статуса UI -> вызов useCases.trainModel(...)
3. Use Case             ──► Проверка payload -> вызов repository.trainModel(...)
4. Repository (HTTP)    ──► POST-запрос на /api/models/train (передача JSON по сети)
   │
   ▼ (Сеть / HTTP)
   БЭКЕНД (Сервер)
5. Роутер               ──► Перехват POST /api/models/train -> вызов контроллера
6. admin.controller.js  ──► Чтение параметров из req.body -> вызов jobService.trainModel(...)
7. job.service.js       ──► [ЯДРО] Проверка условий (надо >= 20 точек),
   вызов математических модулей обучения (MLP / SGD Регрессия).
8. База данных (data/)  ──► Запись обученных весов нейросети в SQLite3.


# 🧠 Архитектура Машинного Обучения и Автокалибровки (ML Pipeline)

В проекте **WiFi Tracker & Speedometer** реализована гибридная двухкомпонентная система машинного обучения. Она разделена между веб-админкой (Бэкенд на Node.js) и автономным мобильным клиентом (Android / TFLite via Python).

Этот документ описывает, как устроены оба конвейера, почему они используют разные математические модели и какие архитектурные компромиссы (плюсы и минусы) заложены в систему.

---

## 🗺️ Общая схема распределения ролей

```text
 ┌────────────────────────────────────────────────────────────────────────┐
 │                      ИСТОРИЧЕСКИЙ ДАТАСЕТ (SQLite3)                    │
 └───────────────────────────────────┬────────────────────────────────────┘
                                     │
            ┌────────────────────────┴────────────────────────┐
            ▼                                                 ▼
┌───────────────────────────────┐                 ┌───────────────────────────────┐
│     ВЕБ-АДМИНКА (Node.js)     │                 │   АНАЛИТИКА / МОБИЛКА (Python) │
│   Локальное обучение «на лету»│                 │   Глобальное тяжелое обучение │
├───────────────────────────────┤                 ├───────────────────────────────┤
│ Модель: Логистический Нейрон  │                 │ Модель: Перцептрон (MLP Keras)│
│ Размерность: 4 фьючи ──► 1    │                 │ Размерность: 7 фьючей ──► 1   │
│ Контекст: Сегмент / Маршрут   │                 │ Контекст: Глобальный (Офлайн) │
└───────────────┬───────────────┘                 └───────────────┬───────────────┘
```

## 1. Конвейер Бэкенда (Панель Администратора на Node.js)

### Как это устроено сейчас
Когда администратор нажимает кнопку «Обучить» в веб-интерфейсе, фронтенд шлет запрос на `/api/models/train`, который обрабатывается в `job.service.js`.
* **Алгоритм:** Однослойная нейросеть — **Логистический нейрон** (логистическая регрессия с сигмоидальной функцией активации), написанная на чистом JavaScript.
* **Входные данные:** 4 агрегированных признака конкретного роутера (`appearanceFrequency`, `avgSignalNorm`, `signalVarianceNorm`, `spatialConsistency`).
* **Логика работы:** Модель обучается локально под геометрию конкретного `segmentId` или `routeId`. Она вычисляет персональную вероятность стабильности (`bssidProbability`) для каждого MAC-адреса вдоль дороги.

### Плюсы и минусы подхода в Ноде

*   **➕ Абсолютная легковесность:** Код написан без использования тяжелых ML-библиотек вроде TensorFlow.js. Проект разворачивается через стандартный `npm install` без компиляции тяжелых C++ бинарников.
*   **➕ Высокая скорость (Zero Blocking):** Обучение на 50–200 локальных роутерах занимает доли миллисекунды. Поток выполнения Node.js (`Event Loop`) не блокируется, сервер продолжает принимать телеметрию с телефонов.
*   **➕ Идеально для малых данных:** На уровне одного дорожного сегмента данных мало. Простой логистический нейрон на малом объеме работает стабильно и защищен от переобучения.
*   **❌ Ограниченная глубина анализа:** Модель не видит скрытых нелинейных зависимостей и не учитывает внешние факторы (время суток, день недели).
*   **❌ Зависимость от сети:** Чтобы очистить трек с помощью этой модели, мобильное приложение обязано иметь стабильный интернет для постоянных запросов к API сервера.

## 2. Конвейер Мобильного приложения (Python ──► Android TFLite)

### Как это устроено сейчас
В папке `tools/nn/` находится независимый Python-скрипт (`train_mlp.py`), использующий библиотеки `TensorFlow/Keras` и `Scikit-learn`.
* **Алгоритм:** Полносвязная нейронная сеть прямого распространения — **Многослойный перцептрон (MLP)** с архитектурой `7 -> 32 -> 16 -> 1` и функциями активации **ReLU** на скрытых слоях.
* **Входные данные:** 7 признаков. К базовым радио-метрикам добавлены временные контексты: `hour_of_day` (час суток) и `day_of_week` (день недели).
* **Логика работы:** Модель обучается крупносерийно (офлайн) на гигантских CSV-выгрузках (`train.csv`, `val.csv`), собранных со всех маршрутов за все время. На выходе она компилируется в бинарный файл `network_stability.tflite` и зашивается в ресурсы Android-приложения.

### Плюсы и минусы мобильного MLP

*   **➕ Полная автономность (Офлайн-режим):** Смартфон выполняет инференс (предсказание) локально на своем процессоре через движок TFLite. Трекер фильтрует зашумленный Wi-Fi и калибрует трек даже в тоннелях, глушилках или зонах без связи.
*   **➕ Учет временных паттернов:** За счет фьючей часа и дня недели MLP понимает, что в пятницу вечером уровень шума и помех в эфире выше, чем в понедельник утром, корректируя строгость фильтра.
*   **➕ Высокая точность:** Архитектура `7->32->16->1` благодаря ReLU-активациям способна выучить сложные нелинейные затухания сигналов (например, проезд через плотную застройку).
*   **❌ Нельзя обучить в админке «на лету»:** Если запустить этот скрипт прямо в Node.js, сервер намертво зависнет на несколько минут из-за тяжелых матричных вычислений на CPU, заблокировав обслуживание клиентов.
*   **❌ Проблема переобучения (Overfitting):** Если скормить этой сети мало данных (например, лог всего одной поездки), она просто «зазубрит» его наизусть, потеряв способность фильтровать сигналы на других участках.

## ⚖️ Архитектурные выводы: Почему они не объединены?

Текущая раздельная реализация — это осознанный **компромисс между гибкостью управления и производительностью**:

1.  **Админка (Node.js) отвечает за оперативность:** Ей не нужен MLP. Задача админки — дать инженеру возможность нажать кнопку и мгновенно обновить список «надежных» локальных роутеров для конкретного перекрестка. С этим идеально справляется быстрый логистический нейрон.
2.  **Скрипт (Python) отвечает за "интеллект" устройства:** Ему не место на веб-сервере. Его задача — переварить терабайты логов, обучить тяжелую нейросетевую модель и выдать оптимизированную «прошивку» (`.tflite`) для мобильных телефонов, чтобы они думали самостоятельно.
